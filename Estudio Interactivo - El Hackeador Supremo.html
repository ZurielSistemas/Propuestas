from pybricks.hubs       import PrimeHub
from pybricks.pupdevices import Motor, ColorSensor
from pybricks.parameters import Button, Direction, Port, Stop
from pybricks.tools      import wait, StopWatch

hub         = PrimeHub()
hub.imu.reset_heading(0)   # Resetear giroscopio al encender

left_motor  = Motor(Port.A, Direction.COUNTERCLOCKWISE)  # Tracción izquierda
right_motor = Motor(Port.E)                                # Tracción derecha
back        = Motor(Port.B, Direction.COUNTERCLOCKWISE)  # Gripper trasero
front       = Motor(Port.F, Direction.COUNTERCLOCKWISE)  # Gripper frontal
back.control.limits(1200, 10000, 250)   # Límites para run_target()
front.control.limits(1200, 10000, 250)

floor = ColorSensor(Port.C)   # Sensor de piso — seguimiento de línea
scan  = ColorSensor(Port.D)   # Sensor lateral — escaneo de bloques
time  = StopWatch()            # Cronómetro global

¿Por qué COUNTERCLOCKWISE en el motor izquierdo?

Los motores A y E están montados en espejo: el eje de uno sale por la izquierda y el del otro por la derecha. Sin la inversión, cuando les das velocidad positiva a los dos, uno empuja hacia adelante y el otro hacia atrás. Con COUNTERCLOCKWISE, ambos empujan en la misma dirección al recibir el mismo número.

Puerto A = tracción izquierda · Puerto E = tracción derecha · Puerto B = gripper trasero · Puerto F = gripper frontal · Puerto C = sensor de piso · Puerto D = sensor de escaneo

control.limits — solo en los grippers

control.limits(1200, 10000, 250) configura el controlador PID interno de los grippers. Solo aplica a motores que usan run_target() (posición precisa). Los motores de tracción usan dc() (voltaje directo) y no pasan por ese controlador — los límites no los afectan.

¿Por qué reset_heading(0) al arrancar?

El giroscopio acumula ángulos entre runs. Si el robot terminó el intento anterior en 270°, al encender el Hub de nuevo ese valor sigue allí. El reset fuerza que 0° sea siempre "recto hacia adelante" al inicio de cada run.

2 Calibrate() — El primer ritual de competencia

def Calibrate():
    hub.speaker.beep(880, 250)    # Beep: "listo para calibrar"
    while(Button.RIGHT in hub.buttons.pressed()):
        pass                         # Esperar que suelten el botón
    hub.speaker.beep(1760, 250)   # Beep: "arrancando calibración"
    wait(500)

    CalibrateHigh = 0
    CalibrateLow  = 100

    calibrateStart = time.time()
    run(40, 40)                       # Avanzar despacio
    while (time.time() - calibrateStart < 1500):   # Durante 1.5 segundos
        CalibrateNow  = floor.reflection()
        CalibrateLow  = min(CalibrateLow,  CalibrateNow)
        CalibrateHigh = max(CalibrateHigh, CalibrateNow)
    stop()

    # Guardar en la memoria permanente del Hub
    hub.system.storage(0, write=bytes([CalibrateLow, CalibrateHigh]))

¿Qué hace exactamente Calibrate()?

El sensor de piso mide qué tan brillante es la superficie (0=negro total, 100=blanco total). El problema: esos valores cambian con la iluminación del lugar. En casa pueden ser 8 y 85. En el gimnasio de competencia, 12 y 92. Calibrate() detecta los valores reales del lugar y los guarda.

Calibrar SIEMPRE en el tapete del lugar de competencia — nunca reutilizar la calibración de casa.

¿Dónde se guardan los valores?

hub.system.storage(0, write=bytes([CalibrateLow, CalibrateHigh])) escribe los dos valores en la memoria permanente del Hub — sobrevive apagados y reinicios. El slot 0 guarda el negro, el slot 1 el blanco. setup() los lee de allí al inicio de cada run.

El robot debe pasar por zonas negras Y blancas durante los 1.5 segundos de calibración — si solo pasa por un color, CalibrateHigh o CalibrateLow quedarán en valores incorrectos.

¿Cómo activar la calibración?

Mantener presionado el botón DERECHO del Hub mientras se enciende. Al soltar, el robot avanza sobre el tapete. Asegurarse de que el recorrido cubra la línea negra y el área blanca del tapete.

3 setup() — Batería y parámetros iniciales
▾ 
def setup():
    BatteryVoltage    = hub.battery.voltage()            # Leer en mV
    BatteryPercentage = round((BatteryVoltage - 7000) / 1.25) / 10
    BatteryScale      = BatteryPercentage / 100          # 0.0 a 1.0

    # Batería llena (8250 mV) → BatteryScale = 1.0
    # Batería media (7625 mV) → BatteryScale = 0.5
    # Batería baja  (7000 mV) → BatteryScale = 0.0

    lightPD    = [0.55, 9]    # PD para seguir línea
    junctionPD = [0.45, 5]    # PD para detectar cruce
    gyroPD     = [6, 30]      # PD para movimiento recto
    minSpeed   = 45 - round(5  * BatteryScale)  # llena: 50 / baja: 45
    accLength  = 65 + round(15 * BatteryScale)  # llena: 80 / baja: 65
    decLength  = 110+ round(15 * BatteryScale)  # llena:125 / baja:110

    # Leer calibración guardada por Calibrate()
    lightLow    = int.from_bytes(hub.system.storage(0, read=1), "big")
    lightHigh   = int.from_bytes(hub.system.storage(1, read=1), "big")
    lightTarget = (lightHigh + lightLow) / 2 - 10  # Promedio, sesgado al negro
    blackThreshold = lightLow  + 5
    whiteThreshold = lightHigh - 8

    heading = 0             # Rumbo inicial: recto
    hub.imu.reset_heading(0)
    time.reset()             # Iniciar cronómetro del run

La fórmula de BatteryScale

La batería varía entre 7000 mV y 8250 mV. La fórmula (voltaje - 7000) / 1.25 / 10 mapea ese rango a 0.0–1.0. Este factor se usa para ajustar velocidades y constantes PD — sin él el robot se comportaría diferente según el nivel de batería.

8.25V → Scale = 1.0  ·  7.63V → Scale = 0.5  ·  7.0V → Scale = 0.0

La conexión con Calibrate()

lightLow y lightHigh vienen exactamente de lo que guardó Calibrate(). Si no se calibró primero, setup() lee valores del intento anterior — potencialmente de otro lugar con otra iluminación. Por eso Calibrate() debe ir siempre antes del primer run.

¿Por qué lightTarget = promedio − 10?

Si el target fuera el promedio exacto, el robot seguiría el centro de la línea. Al restar 10, el punto de seguimiento se mueve hacia el negro — el robot "prefiere" estar sobre el lado oscuro. Esto lo hace más robusto ante sombras de jueces, ondulaciones del tapete o variaciones de luz en competencia.

4 getYaw() — El giroscopio y el problema del salto angular
▾ 
def getYaw():
    global yaw
    yaw = hub.imu.heading() * gyroRatio
    # gyroRatio corrige si el Hub sobre/subestima ángulos reales

    yaw = yaw - round((yaw - heading) / 370) * 360
    # ANTI-WRAPAROUND: evita el salto de +180° a -180°

def resetYaw():
    global heading
    hub.imu.reset_heading(0)
    heading = 0

# Valores típicos de gyroRatio:
# gyroRatio = 1.0   → giroscopio normal (usar por defecto)
# gyroRatio = 0.98  → Hub gira de más  (leer 92° cuando giró 90°)
# gyroRatio = 1.02  → Hub gira de menos (leer 88° cuando giró 90°)

El problema del salto angular

El giroscopio acumula ángulos — si el robot hace varios giros, puede llegar a valores como 375° o 450°. Si el heading objetivo es 15°, la diferencia real es 0°, pero 375 - 15 = 360°. El controlador PD vería un error enorme y haría girar el robot en círculos.

yaw = yaw - round((yaw - heading) / 370) * 360
Esta fórmula detecta cuántas vueltas completas de diferencia hay entre yaw y heading, y las resta. Yaw siempre queda en el rango correcto. Se usa 370 (no 360) para evitar ambigüedad en el borde exacto de 180°.

¿Qué es gyroRatio y cómo ajustarlo?

Algunos Hubs leen ángulos ligeramente diferentes a los reales. Si el Hub dice 92° cuando el robot giró 90° físicamente, todos los giros acumularán error.

Prueba: haz 4 veces turn(3, 90). El robot debe quedar exactamente donde empezó. Si queda desviado a la derecha: gira de más → bajar gyroRatio a ~0.98. Si queda desviado a la izquierda: subir a ~1.02. Ajustar en pasos de 0.01.

¿Qué es "heading" como variable global?

heading es el rumbo objetivo del robot — siempre empieza en 0 (recto hacia adelante). Cada vez que el robot ejecuta turn(), heading se actualiza sumando el cambio de ángulo. getYaw() compara el ángulo real del giroscopio con este objetivo para calcular la corrección.

5 Control PD — El corazón del movimiento
▾ 
def PD(input1, input2):
    global lastError, PDscale
    PDscale   = (abs(power) / 100) * 0.7 + 0.3  # Entre 0.3 y 1.0
    error     = input1 - input2                   # Diferencia: objetivo - actual
    result    = (error * kP) + ((error - lastError) * kD)
    result    = result * PDscale
    lastError = error
    return result

def setPD(PD, reset):
    global kP, kD, lastError
    kP, kD = PD[0], PD[1]
    if reset == 1: lastError = 0   # reset=0 mantiene el error anterior (transición suave)

def acceleration(movedDistance, totalDistance, targetSpeed, acc, dec):
    speedChange = abs(targetSpeed) - minSpeed
    acc_speed = speedChange * (movedDistance / accLength) + minSpeed
    dec_speed = speedChange * ((totalDistance - movedDistance) / decLength) + minSpeed
    return min(acc_speed, dec_speed, abs(targetSpeed))
    # min() siempre toma la curva más baja → perfil trapecial suave

Proporcional (P) — reacciona al error actual

error × kP: si el robot se desvió 10°, aplica corrección proporcional a 10. Problema: con solo P el robot llega al objetivo pero lo sobrepasa y oscila — se corrige de más, vuelve, se corrige de más otra vez.

Derivativo (D) — reacciona al cambio del error

(error - lastError) × kD: mide qué tan rápido está cambiando el error. Si el error está disminuyendo rápido (el robot se está corrigiendo agresivamente), D aplica un freno para que no sobrepase. Es la amortiguación del sistema.

P empuja hacia el objetivo. D frena para no pasarse. Juntos logran llegada rápida y precisa sin oscilación.

PDscale — corrección dinámica por velocidad

Cuando el robot va lento (power bajo), la corrección se reduce automáticamente. A velocidad 0: PDscale = 0.3. A velocidad 100: PDscale = 1.0. Evita correcciones violentas durante el arranque y el frenado, donde una corrección exagerada haría que el robot saliese de la línea.

Aceleración trapezoidal

El robot sube suavemente de minSpeed a targetSpeed (rampa de aceleración), mantiene velocidad, y baja suavemente al acercarse al destino (rampa de frenado). min() siempre toma la más baja de las tres curvas — en distancias cortas los perfiles se solapan y el robot nunca supera la velocidad que corresponde.

# ── move() ───────────────────────────────────────────────────────
def move(speed, totalDistance, acc=True, dec=True):
    resetMotors()
    movedDistance = 0
    setPD(gyroPD, 1)
    while(movedDistance <= totalDistance):
        movedDistance = (abs(left_motor.angle()) + abs(right_motor.angle())) / 2
        power = acceleration(movedDistance, totalDistance, speed, acc, dec)
        power = round((abs(speed)/speed) * power)   # Conservar el signo
        getYaw()
        correction = round(PD(yaw, heading))
        run(power - correction, power + correction)
    stop()

# ── moveWhite / moveBlack ─────────────────────────────────────────
def moveWhite(speed):   # Avanza hasta encontrar superficie brillante
    while(floor.reflection() < whiteThreshold):
        getYaw(); correction = round(PD(yaw, heading))
        run(speed - correction, speed + correction)
    stop()

def moveBlack(speed):   # Avanza hasta encontrar superficie oscura
    while(floor.reflection() > blackThreshold):
        getYaw(); correction = round(PD(yaw, heading))
        run(speed - correction, speed + correction)
    stop()

# ── turn() ────────────────────────────────────────────────────────
def turn(mode, change, speed=100, checking=15):
    heading = heading + change
    # kP baja con batería llena (motores más fuertes → no sobrepasar)
    # kD sube con batería llena (más amortiguación necesaria)
    if mode <= 2:
        setPD([15 - round(3*BatteryScale), 180 + round(50*BatteryScale)], 1)
    else:
        setPD([12 - round(3*BatteryScale), 220 + round(50*BatteryScale)], 1)
    count = 0
    while(count <= checking):
        getYaw()
        correction = round(PD(yaw, heading))
        if   mode == 1: run(-correction, 0)          # Pivote izquierda
        elif mode == 2: run(0, correction)           # Pivote derecha
        else:           run(-correction, correction)  # Giro en sitio
        count = count + 1 if abs(yaw - heading) <= 2.5 else 0
    stop()

# ── follow / followUntilJunction ─────────────────────────────────
def follow(side, speed, totalDistance, acc=True, dec=True, reset=False):
    setPD(lightPD, 1)
    lightFiltered = floor.reflection()
    while(movedDistance <= totalDistance):
        raw = floor.reflection()
        lightFiltered = (lightFiltered * 3 + raw) / 4  # Filtro promedio móvil
        correction = PD(lightTarget, lightFiltered)
        if side == 2: correction *= -1
        run(power - correction, power + correction)
    stop()

def followUntilJunction(side, speed):
    setPD(junctionPD, 0)   # reset=0 → transición suave desde follow()
    while(floor.reflection() > blackThreshold):
        correction = PD(lightTarget, floor.reflection())
        if side == 2: correction *= -1
        run(speed - correction, speed + correction)
    stop()

move() — distancia en grados de rueda

360° de rueda ≈ 19.6 cm con la rueda grande del robot. El código promedia ambos encoders para que si una rueda patina o la corrección gyro la hace girar más, el error se distribuya. Ejemplo: move(100, 720) ≈ avanzar 39 cm.

moveWhite y moveBlack — navegación por sensor

En vez de distancia fija, el robot avanza hasta encontrar un color específico. Esto lo hace robusto ante pequeñas variaciones de posición inicial — si el robot arrancó 1 cm adelante del lugar ideal, una distancia fija lo posicionaría mal, pero moveBlack() siempre para exactamente sobre la línea.

turn() — tres modos de giro

  • mode=1: solo rueda izquierda — gira sobre la derecha (radio grande)
  • mode=2: solo rueda derecha — gira sobre la izquierda (radio grande)
  • mode=3: ambas en sentido contrario — giro en sitio (más rápido, más preciso)
checking=15 obliga al robot a estar dentro de ±2.5° durante 15 ciclos consecutivos antes de parar. Evita que pare mientras todavía tiene inercia y está oscilando.

follow() — el filtro de promedio móvil

(lightFiltered × 3 + raw) / 4 promedia la lectura nueva con 3 veces la anterior. Suaviza el ruido del sensor — sin esto, una variación brusca (sombra de un juez, polvo en el tapete) causaría una corrección violenta que sacaría al robot de la línea.

followUntilJunction vs follow

followUntilJunction usa junctionPD más conservador que lightPD — menos corrección agresiva para que el robot no pase de largo el cruce. También llama setPD(..., 0) sin resetear lastError, para que la transición desde un follow() previo sea fluida.

def backDegree(speed, angle, pause=False):
    # angle: 0=cerrado, 100=abierto
    back.run_target(abs(round(speed * 12)), angle * -1, wait=pause)

def frontDegree(speed, angle, pause=False):
    # angle: 0=abierto, 100=cerrado
    front.run_target(abs(round(speed * 12)), abs(angle), wait=pause)

# Ejemplo de uso en paralelo (pause=False — default):
front.dc(-40)             # Abrir frontal directo (voltaje, sin esperar)
back.dc(40)              # Cerrar trasero directo (voltaje, sin esperar)
move(100, 350, dec=False) # ← Esto se ejecuta MIENTRAS los grippers se mueven

# Ejemplo de uso secuencial (pause=True — esperar posición):
backDegree(100, 55, pause=True)  # Espera a que llegue a 55° antes de continuar
move(-100, 150)                  # Ahora sí avanza

run_target vs dc — dos formas de mover un motor

motor.dc(velocidad): control directo de voltaje. El motor gira a esa potencia sin saber ni importarle en qué ángulo está. Rápido pero sin precisión de posición — se usa para mover rápido sin importar dónde para.

motor.run_target(velocidad, ángulo, wait=...): el motor usa su controlador PID interno para llegar exactamente al ángulo indicado y mantenerse allí. Preciso y controlado — usa para posicionamientos exactos de agarre.

control.limits(1200, 10000, 250) configura el controlador interno de run_target: velocidad máxima, aceleración máxima, torque. Sin esto, run_target puede moverse demasiado despacio o ser demasiado brusco.

pause=False — ejecución en paralelo

Cuando pause=False (default), el código llama a run_target y continúa inmediatamente a la siguiente línea sin esperar que el gripper llegue. Esto permite que el robot avance mientras el gripper se mueve simultáneamente — ahorra tiempo en las misiones.

Si el gripper no llega a posición antes de que el robot intente agarrar el bloque, usar pause=True o agregar wait() para darle tiempo suficiente.

¿Por qué angle × -1 en backDegree?

El motor B tiene COUNTERCLOCKWISE como dirección. Para que "0=cerrado, 100=abierto" sea la misma convención física independientemente del montaje, se invierte el signo del ángulo. Así el código que llama backDegree(100, 0) siempre significa "cerrar el gripper trasero".

speed × 12 — conversión de escala

run_target acepta velocidad en grados por segundo. Multiplicar el parámetro de "0 a 100%" por 12 convierte a la escala interna — speed=100 → 1200 °/s, que es el límite configurado en control.limits.

def readColour():
    global h, s, v

    for section in range(2):         # 2 secciones del tapete
        for blocks in range(2):      # 2 bloques por sección

            follow(2, 100, 500, acc=False, dec=False)  # Cruzar a otra sección
        else:

El offset de 20° después del cruce — detalle crítico

Después de followUntilJunction, el código hace move(90, 20) antes de leer. Esto es porque el robot detecta el cruce cuando el sensor C (piso, frente-centro) ve negro, pero el sensor D (lateral) está físicamente más adelante del cuerpo del robot. Sin el offset, el sensor D escanearía el bloque equivocado.

Detectar la línea ≠ estar en posición de escaneo. Los 20° compensan la distancia física entre el sensor de detección y el sensor lateral.

¿Qué es HSV?

  • H (Hue): el tono de color — 0°=rojo, 60°=amarillo, 120°=verde, 240°=azul
  • S (Saturation): intensidad — 0=gris/blanco/negro sin color, 100=color puro vivo
  • V (Value): brillo — 0=negro total, 100=blanco brillante

La lógica de clasificación de ProcessColour()

Paso 1 — Alta saturación: Azul y amarillo son colores vivos → S alto. Se toman los 2 índices con mayor S. Dentro de ellos, el que tiene mayor H es azul (H≈240°), el otro es amarillo (H≈60°).

Paso 2 — Baja saturación: Blanco y negro no tienen color definido → S bajo. Se toman los 2 con menor S. El que tiene mayor V (más brillante) es blanco, el otro es negro.

Paso 3 — Ajuste de índices: Como verde está fijo en posición 2 y rojo en posición 3, cualquier índice mayor a 1 debe ajustarse sumando 2 para compensar esas posiciones fijas.

Esta función está marcada "MODIFY AT YOUR OWN RISK". Si se modifica sin entenderla completamente, todo el sistema de notas puede fallar. Entender primero, modificar después.

¿Por qué este orden de misiones?

Las misiones de notas musicales (bloques de colores) se agrupan al inicio porque están todas en la misma zona del tapete. Separarlas de los cables causaría recorridos innecesarios. El orden minimiza la distancia total recorrida dentro del límite de 2 minutos.

Navegación dinámica — currentPosition y currentDirection

El robot no tiene un camino fijo. TakeBlueWhite y TakeBlackYellow usan currentPosition/currentDirection para adaptarse a dónde quedaron los bloques aleatorios. La lógica de target[] y taking decide la ruta óptima en tiempo real.

Parámetros críticos a ajustar para tu tapete

  • angleX = 25 en Put1() — ángulo de depósito del primer grupo. Ajustar si los bloques no quedan en la posición correcta.
  • redAngle = 39 en Put2() — ángulo hacia la caja roja desde el cruce. Crítico: si está mal, rojo/negro/amarillo van al lugar equivocado.
  • underturn = 5 en Instruments() — ajuste fino de la aproximación final al escenario.

El selector de botones — el programa no empieza en Main()

Al encender, el Hub ejecuta el código desde arriba. Las últimas líneas del archivo detectan qué botón está presionado: Botón DERECHO → Calibrate(). Botón IZQUIERDO → limpiar ruedas (correr sin parar). Sin botón → Main(). Esto es el "menú" del robot.

El sistema de batería

BatteryVoltage    = hub.battery.voltage()                    # Lee voltaje en mV (ej: 8100)
BatteryPercentage = round((BatteryVoltage - 7000) / 1.25) / 10  # Convierte a porcentaje
BatteryScale      = BatteryPercentage / 100                  # Factor 0.0 a 1.0
if BatteryScale < 0:
    BatteryScale = 0                                         # No puede ser negativo

¿Por qué (voltaje - 7000) / 1.25 / 10?

La batería LiPo del Hub Spike Prime trabaja entre 7000 mV (agotada) y 8250 mV (llena).

La fórmula mapea ese rango a 0–100%:

  • 7000 mV → (7000-7000)/1.25/10 = 0% → BatteryScale = 0.0
  • 8250 mV → (8250-7000)/1.25/10 = 100% → BatteryScale = 1.0
  • 7625 mV → 50% → BatteryScale = 0.5

¿Para qué sirve BatteryScale?

A mayor voltaje, los motores giran más rápido con el mismo comando de potencia.

Si no compensas, el robot se comporta diferente con batería llena vs. batería baja.

BatteryScale permite ajustar velocidades y constantes automáticamente.

Es una forma de representar colores con 3 números:

  • S (Saturation / Saturación): qué tan "intenso" es el color — 0=gris, 100=color puro

Por eso HSV es mejor que RGB para clasificar colores:

  • Azul: H≈240, S alto, V alto
  • Amarillo: H≈60, S alto, V alto
  • Blanco: S≈0, V alto (no importa H)
  • Negro: S≈0, V bajo

6 El sistema PD — el corazón del movimiento
▼ 
def setPD(PD, reset):
    global kP, kD, lastError
    kP = PD[0]
    kD = PD[1]
    if reset == 1:
        lastError = 0

¿Qué son kP y kD?

Son las "ganancias" del controlador PD:

  • kP (Proporcional): qué tan fuerte reacciona el robot al error actual
  • kD (Derivativo): qué tan fuerte reacciona al cambio del error

lastError guarda el error de la iteración anterior — necesario para calcular

el componente D. Se resetea a 0 cuando empieza un nuevo movimiento.

def PD(input1, input2):
    global lastError, PDscale
    PDscale = (abs(power) / 100) * 0.7 + 0.3   # Entre 0.3 y 1.0
    error = input1 - input2                      # Diferencia entre objetivo y actual
    result = (error * kP) + ((error - lastError) * kD)  # Cálculo PD
    result = result * PDscale                    # Escalar según velocidad
    lastError = error                            # Guardar para la próxima iteración
    return result

La fórmula PD explicada:

error = input1 - input2

Por ejemplo, en seguimiento de línea: lightTarget - luz_actual.

Si el sensor lee menos luz de la que debería → error positivo → corregir a la izquierda.

error * kP = componente Proporcional

Corrección directamente proporcional al error. Si el error es grande, corrige fuerte.

Problema: si solo usas P, el robot llega al objetivo pero lo sobrepasa y oscila.

(error - lastError) * kD = componente Derivativo

Mide qué tan rápido está cambiando el error. Si el error está disminuyendo rápido

(el robot se está corrigiendo), D frena esa corrección para no sobrepasar.

Es como la amortiguación de un carro: sin ella, el carro rebotaría en cada bache.

PDscale = (abs(power) / 100) * 0.7 + 0.3

Cuando el robot va lento (poco power), la corrección se reduce. A velocidad 0:

PDscale = 0.3 (corrección mínima). A velocidad 100: PDscale = 1.0 (corrección completa).

Evita que el robot "luche" con correcciones fuertes cuando va casi detenido.

def acceleration(movedDistance, totalDistance, targetSpeed, acc, dec):
    speedChange = abs(targetSpeed) - minSpeed
    if acc:
        acc_speed = speedChange * (movedDistance / accLength) + minSpeed
    else:
        acc_speed = abs(targetSpeed)

    if dec:
        dec_speed = speedChange * ((totalDistance - movedDistance) / decLength) + minSpeed
    else:
        dec_speed = abs(targetSpeed)

    speed = min(acc_speed, dec_speed, abs(targetSpeed))
    return speed

Perfil de aceleración trapezoidal:

En vez de ir de 0 a 100 de golpe (lo que haría patinar las ruedas y perder tracción),

el robot sube gradualmente de minSpeed a targetSpeed, y luego baja gradualmente

al acercarse al final.

Durante la aceleración: speed = minSpeed + (targetSpeed - minSpeed) * (distancia_recorrida / accLength)

Cuando movedDistance == accLength, la velocidad alcanza targetSpeed.

min(acc_speed, dec_speed, targetSpeed) toma siempre la más baja de las tres curvas.

Esto hace que cuando la distancia total es corta y las curvas de aceleración y frenado

se solapan, el robot nunca supere la velocidad que debería tener.

        run(power - correction, power + correction)

¿Cómo mide la distancia?

Suma los ángulos de los dos encoders y divide entre 2 (promedio).

Así, si una rueda patina y gira más que la otra, el error se promedia.

360° de rueda ≈ 19.6 cm (depende del diámetro de la rueda).

¿Cómo corrige el rumbo?

getYaw() lee el giroscopio → PD(yaw, heading) calcula cuánto se desvió

→ la corrección se suma a una rueda y se resta a la otra.

Si la corrección es positiva (robot se desvió a la derecha):

  • Rueda izquierda: power - correction (más lenta)
  • Rueda derecha: power + correction (más rápida)

→ el robot gira a la izquierda para volver al rumbo.

acc=False / dec=False: se puede desactivar la aceleración o el frenado.

Se usa cuando hay movimientos encadenados y no quieres frenar entre ellos.

Mismo principio que move() pero en vez de medir distancia, mide tiempo.

En el modo 3 (ambas ruedas), sync sincroniza que ambas ruedas giren la misma

cantidad. Si una rueda avanza más que la otra, sync la frena ligeramente.

sync *= 1 es un placeholder — el coeficiente se puede ajustar si hay desequilibrio.

if abs(correction) > speed: limita la corrección máxima al valor de speed.

Evita que el controlador PD ordene más velocidad de la permitida.

follow() — seguir línea con sensor de luz

def follow(side, speed, totalDistance, acc=True, dec=True, limit=False, reset=False):
    global power, heading
    resetMotors()
    movedDistance = 0
    setPD(lightPD, 1)
    raw = floor.reflection()
    lightFiltered = raw
    while(movedDistance <= totalDistance):
        movedDistance = (abs(left_motor.angle()) + abs(right_motor.angle())) / 2
        power = acceleration(...)
        raw = floor.reflection()
        lightFiltered = (lightFiltered * 3 + raw) / 4    # Filtro de promedio móvil
        light = lightFiltered
        correction = PD(lightTarget, light)
        if side == 2:
            correction = correction * -1
        run(power - correction, power + correction)
        wait(3)
    stop()
    if reset:
        resetYaw()

side=1 vs side=2:

El sensor sigue el BORDE de la línea, no el centro. Hay dos bordes posibles:

  • side=1: borde izquierdo de la línea negra (sensor a la izquierda de la línea)
  • side=2: borde derecho (sensor a la derecha) → se invierte la corrección

El filtro (lightFiltered * 3 + raw) / 4:

Promedia la lectura actual con 3 veces la lectura anterior. Esto suaviza el ruido

del sensor — si hay una variación brusca (sombra, polvo), no afecta tanto.

Si el filtro fuera solo raw, pequeñas variaciones causarían correcciones bruscas.

wait(3): espera 3ms entre iteraciones. Da tiempo al sensor para hacer una

lectura estable. Sin este wait, el loop correría tan rápido que las lecturas

serían menos confiables.

if reset: resetYaw(): después del seguimiento de línea, el giroscopio puede

haber acumulado algo de deriva. resetYaw() lo limpia para el siguiente movimiento giroscópico.

followUntilJunction() — seguir línea hasta un cruce

def followUntilJunction(side, speed):
    global power, heading
    setPD(junctionPD, 0)    # PD diferente, sin resetear lastError
    power = speed
    light = floor.reflection()
    while(light > blackThreshold):    # Mientras no encuentre negro del cruce
        light = floor.reflection()
        correction = PD(lightTarget, light)
        if abs(correction) >= correctionLimit:
            correction = correctionLimit * (correction / abs(correction))
        if side == 2:
            correction = correction * -1
        run(power - correction, power + correction)
        wait(3)
    stop()

¿Por qué junctionPD diferente a lightPD?

Al llegar a un cruce, el robot necesita detectar negro muy rápido y parar.

junctionPD = [0.45, 5] es más conservador que lightPD = [0.55, 9]

para evitar que el robot pase de largo el cruce por ir corrigiendo agresivamente.

setPD(junctionPD, 0) — nota el 0 en lugar de 1:

No resetea lastError. Así el D recuerda el error anterior y la transición

desde un follow() previo es suave.

correctionLimit: limita la corrección máxima a 10 unidades. En cruces,

el sensor puede leer valores extremos (línea muy negra) que darían correcciones

enormes. Este límite previene giros bruscos en el cruce.

def ProcessColour():
    global blue, white, yellow, black, green, red
    green = 2  # Siempre en posición 2
    red   = 3  # Siempre en posición 3

    # Los de mayor saturación son azul y amarillo (colores puros)
    top2 = sorted(range(len(s)), key=lambda i: (-s[i], i))[:2]
    test1, test2 = top2[0], top2[1]
    blue, yellow = (test1, test2) if h[test1] > h[test2] else (test2, test1)

    # Los de menor saturación son blanco y negro (sin color)
    low2 = sorted(range(len(s)), key=lambda i: (s[i], i))[:2]
    test1, test2 = low2[0], low2[1]
    white, black = (test1, test2) if v[test1] > v[test2] else (test2, test1)

    # Ajustar índices (las posiciones 1 y 2 son bloques escaneados 0 y 1,
    # pero hay que saltar las posiciones fijas 2 y 3 en el tapete)
    blue, white, yellow, black = (
        x + 2 if x > 1 else x
        for x in (blue, white, yellow, black)
    )

La lógica de clasificación — paso a paso:

Paso 1: Separar por saturación.

Azul y amarillo tienen saturación ALTA (colores vivos).

Blanco y negro tienen saturación BAJA (sin color definido).

top2 = los 2 índices con mayor S → son azul o amarillo.

low2 = los 2 índices con menor S → son blanco o negro.

Paso 2: Dentro de azul/amarillo, separar por Hue.

Amarillo tiene H ≈ 60, Azul tiene H ≈ 240. El mayor H es azul.

if h[test1] > h[test2]: el primero es azul, el segundo es amarillo.

Paso 3: Dentro de blanco/negro, separar por Value (brillo).

Blanco tiene V alto, Negro tiene V bajo.

if v[test1] > v[test2]: el más brillante es blanco.

Paso 4: Ajustar índices.

Los bloques escaneados están en las posiciones 1, 2, 5, 6 del tapete (con verde=2 y rojo=3).

Los índices 0 y 1 de las listas corresponden a posiciones 1 y 2.

Los índices 2 y 3 corresponden a posiciones 5 y 6.

Como verde y rojo ocupan las posiciones 2 y 3, los índices 2 y 3 de los escaneados

se convierten en posiciones 4 y 5 → se suma 2 si el índice > 1.

def getYaw():

7 El sistema PD — el corazón del movimiento
▼ 
def setPD(PD, reset):
    global kP, kD, lastError
    kP = PD[0]
    kD = PD[1]
    if reset == 1:
        lastError = 0

¿Qué son kP y kD?

Son las "ganancias" del controlador PD:

  • kP (Proporcional): qué tan fuerte reacciona al error actual
  • kD (Derivativo): qué tan fuerte reacciona al cambio del error

if reset == 1: lastError = 0 — resetea el error previo para transición limpia.

Se llama con reset=0 cuando se quiere transición suave (como al pasar de

follow() a followUntilJunction()).

def PD(input1, input2):
    global lastError, PDscale
    PDscale = (abs(power) / 100) * 0.7 + 0.3   # Entre 0.3 y 1.0
    error   = input1 - input2
    result  = (error * kP) + ((error - lastError) * kD)
    result  = result * PDscale
    lastError = error
    return result

La fórmula PD explicada:

error = input1 - input2 — diferencia entre objetivo y actual.

En seguimiento de línea: lightTarget - luz_actual.

En movimiento recto: yaw - heading.

error * kP = componente Proporcional

Corrección proporcional al error. Si el error es grande, corrige fuerte.

Problema solo con P: el robot llega al objetivo pero lo sobrepasa y oscila.

(error - lastError) * kD = componente Derivativo

Mide qué tan rápido cambia el error. Si el error está bajando rápido

(el robot se está corrigiendo), D frena esa corrección para que no sobrepase.

Es la amortiguación del sistema.

PDscale = (abs(power)/100) * 0.7 + 0.3 — escala la corrección según la velocidad.

A velocidad 0: PDscale = 0.3 (corrección mínima).

A velocidad 100: PDscale = 1.0 (corrección completa).

Evita correcciones violentas durante arranque y frenado.

def acceleration(movedDistance, totalDistance, targetSpeed, acc, dec):
    speedChange = abs(targetSpeed) - minSpeed
    if acc:
        acc_speed = speedChange * (movedDistance / accLength) + minSpeed
    else:
        acc_speed = abs(targetSpeed)

    if dec:
        dec_speed = speedChange * ((totalDistance - movedDistance) / decLength) + minSpeed
    else:
        dec_speed = abs(targetSpeed)

    speed = min(acc_speed, dec_speed, abs(targetSpeed))
    return speed

Aceleración trapezoidal:

El robot no arranca de golpe a velocidad máxima — sube gradualmente.

  • acc_speed: sube de minSpeed a targetSpeed proporcional a movedDistance
  • dec_speed: baja de targetSpeed a minSpeed proporcional a la distancia restante
  • min(acc_speed, dec_speed, targetSpeed): siempre toma el menor de los tres

En distancias cortas, las dos curvas se solapan — el robot nunca llega a targetSpeed

y hace una curva triangular suave. En distancias largas hace el perfil trapezoidal

completo: sube, mantiene velocidad máxima, baja.

acc=False o dec=False desactivan la rampa correspondiente para movimientos

que deben ser inmediatos (por ejemplo, posicionamiento fino después de un cruce).

def move(speed, totalDistance, acc=True, dec=True):
    global power
    resetMotors()
    movedDistance = 0
    setPD(gyroPD, 1)
    while(movedDistance <= totalDistance):
        movedDistance = (abs(left_motor.angle()) + abs(right_motor.angle())) / 2
        power = acceleration(movedDistance, totalDistance, speed, acc, dec)
        power = round((abs(speed) / speed) * power)
        getYaw()
        correction = round(PD(yaw, heading))
        run(power - correction, power + correction)
    stop()

¿Por qué promediar los dos encoders?

Si el robot está corrigiendo el rumbo (un motor más rápido que el otro),

uno registra más grados que el otro. Usar solo uno daría una distancia incorrecta.

El promedio (abs(left) + abs(right)) / 2 refleja la distancia lineal real del centro.

(abs(speed)/speed) * power — preserva el signo original de speed.

Si speed = -100 (ir hacia atrás), acceleration() devuelve un valor positivo,

y esta línea lo convierte de vuelta a negativo.

La corrección giroscópica:

run(power - correction, power + correction) — si el robot se desvió a la derecha

(yaw > heading, correction positiva), disminuye la rueda izquierda y aumenta la derecha

para corregir hacia la izquierda. Siempre corrige sumando a una rueda y restando a la otra.

def follow(side, speed, totalDistance, acc=True, dec=True, limit=False, reset=False):
    global power, heading
    resetMotors()
    movedDistance = 0
    setPD(lightPD, 1)
    raw = floor.reflection()
    lightFiltered = raw
    while(movedDistance <= totalDistance):
        movedDistance = (abs(left_motor.angle()) + abs(right_motor.angle())) / 2
        LFmovedDistance = movedDistance - LFslowDistance
        if LFmovedDistance < 0:
            LFmovedDistance = 0
        power = acceleration(LFmovedDistance, totalDistance - LFslowDistance, speed, acc, dec)
        raw = floor.reflection()
        lightFiltered = (lightFiltered * 3 + raw) / 4   # Filtro promedio móvil
        light = lightFiltered
        correction = PD(lightTarget, light)
        if limit and abs(correction) >= correctionLimit:
            correction = correctionLimit * (correction / abs(correction))
        if side == 2:
            correction = correction * -1
        run(power - correction, power + correction)
        wait(3)
    stop()
    if reset:
        resetYaw()

side: en qué borde de la línea sigue el robot.

● Nivel intermedio — Rafael debe dominar completamente
I1 ¿Qué hace el componente Proporcional del PD? ¿Y el Derivativo? ¿Por qué se necesitan los dos?
Proporcional (P): corrige el error actual. Si el robot se desvió 10°, aplica corrección de 10×kP. Problema: solo con P, el robot sobrepasa el objetivo y oscila indefinidamente.

Derivativo (D): corrige la velocidad de cambio del error. Si el error está disminuyendo rápido (el robot se está corrigiendo agresivamente), D aplica un freno para que no sobrepase. Es la amortiguación del sistema.

Juntos logran llegada rápida y precisa sin oscilación — P lleva al objetivo, D frena para no pasarse.
I2 ¿Por qué lightTarget = promedio − 10 y no el promedio exacto entre negro y blanco?
Si el target fuera el promedio exacto, el robot seguiría el centro de la línea negra. Al restar 10, el punto de seguimiento se mueve hacia el negro — el robot "prefiere" estar un poco sobre el lado oscuro. Esto lo hace más robusto ante sombras de jueces, ondulaciones del tapete o variaciones de luz en competencia: ya estaba sesgado hacia donde la línea es más fácil de detectar.
I3 ¿Qué hace gyroRatio y cómo sabes si necesitas ajustarlo?
gyroRatio compensa si el Hub sobreestima o subestima los ángulos reales. Algunos Hubs registran 92° cuando el robot giró 90° físicamente — acumulado en muchos giros, esto causa derivas importantes.

Cómo verificar: haz 4 veces turn(3, 90). El robot debe quedar exactamente donde empezó. Si queda desviado a la derecha → gira de más → bajar gyroRatio a ~0.98. Si queda desviado a la izquierda → subir a ~1.02. Ajustar en pasos de 0.01.
I4 ¿Por qué followUntilJunction usa un PD diferente al de follow?
Al llegar a un cruce, el robot necesita detectar negro y parar rápidamente. junctionPD = [0.45, 5] es más conservador que lightPD = [0.55, 9] — menos agresivo en la corrección para que el robot no pase de largo el cruce por corregir demasiado fuerte justo en el momento crítico.

Además, se llama con setPD(..., 0) sin resetear lastError, para que la transición desde el follow() previo sea suave en lugar de abrupta.
I5 ¿Qué significa pause=False en backDegree() y cuándo usarías pause=True?
pause=False (default): el código llama a run_target y continúa inmediatamente sin esperar que el gripper llegue. El gripper se mueve en paralelo con el movimiento del robot — ahorra tiempo valioso en las misiones.

pause=True: el código espera a que el motor llegue exactamente a la posición antes de continuar. Usar cuando el robot necesita que el gripper esté en posición antes de avanzar hacia el bloque, para no chocar con él o fallar el agarre.
I6 ¿Por qué move() promedia los dos encoders en vez de usar solo uno?
Si el robot está girando para corregir el rumbo con el PD giroscópico, una rueda avanza más que la otra. Si usara solo el encoder izquierdo y ese motor está corrigiendo menos (más lento), el robot avanzaría más de la distancia esperada antes de parar.

Al promediar (abs(left) + abs(right)) / 2, los efectos de la corrección se cancelan mutuamente — el promedio refleja la distancia lineal real del centro del robot, independientemente de cuánto corrija el PD en cada ciclo.
● Nivel avanzado — para impresionar a los jueces
A1 ¿Qué es el anti-wraparound en getYaw() y por qué es necesario?
El giroscopio acumula ángulos — si el robot hace varios giros puede llegar a valores como 375° o 450°. Si el heading objetivo es 15°, la diferencia real es 0°, pero 375 - 15 = 360°. El controlador PD vería un error enorme y haría girar el robot en círculos.

La fórmula yaw - round((yaw - heading) / 370) × 360 detecta cuántas vueltas completas de diferencia hay entre yaw y heading, y las resta. Así yaw siempre queda en el rango correcto relativo al heading. Se usa 370 en vez de 360 para evitar ambigüedad en el borde exacto de 180°.
A2 Explica cómo ProcessColour() distingue azul de amarillo usando HSV.
Paso 1: separar los 4 bloques escaneados en dos grupos por saturación (S). Azul y amarillo son colores puros → S alto. Blanco y negro no tienen color definido → S bajo. top2 toma los 2 índices con mayor S.

Paso 2: dentro del grupo de alta saturación, separar por Hue (H). Amarillo tiene H ≈ 60°. Azul tiene H ≈ 240°. El que tiene mayor H es azul: blue, yellow = (test1, test2) if h[test1] > h[test2].

Paso 3: ajustar índices sumando 2 a los que están después de las posiciones fijas de verde (2) y rojo (3).
A3 ¿Por qué el giro tiene kP más bajo con batería llena?
Con batería llena los motores responden con más fuerza al mismo comando. El robot tiende a sobrepasar el ángulo objetivo — la inercia es mayor. Para compensar:

- kP más bajo: menos agresividad en la corrección → no empuja tan fuerte al acercarse al objetivo
- kD más alto: más amortiguación → frena el exceso de inercia más efectivamente

Con batería baja ocurre lo contrario: motores débiles, tienden a quedarse cortos → kP más alto y kD más bajo para que corrija con más fuerza.
A4 ¿Para qué sirve el offset de 20° después de followUntilJunction en Scan()?
El robot detecta el cruce cuando el sensor C (piso, debajo del centro) ve negro. Pero el sensor D (lateral, escaneo) está físicamente más adelante en el chasis.

Sin el move(90, 20), el sensor D escanearía todavía frente al bloque anterior, no frente al bloque que acaba de detectar el cruce. Los 20° (~1.1 cm) compensan exactamente la distancia física entre los dos sensores — es como hacer un "paso extra" para que el sensor correcto quede frente al objeto correcto.
A5 ¿Cómo sabe el robot en TakeBlueWhite() si el próximo bloque está más cerca o más lejos de casa?
Después de recoger el primer bloque, el código compara target[taking] (posición del siguiente bloque a recoger) con currentPosition (dónde está el robot ahora):

- Si target[taking] > currentPosition → el siguiente está más lejos de casa → currentDirection = 1 (avanzar alejándose)
- Si target[taking] < currentPosition → está más cerca → currentDirection = -1 (retroceder hacia casa)

Luego turn(3, -90 * currentDirection) gira al lado correcto para ir en esa dirección.
